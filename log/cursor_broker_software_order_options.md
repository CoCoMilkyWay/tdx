## 核心矛盾与解法

"客户端还在运行" + "脚本独立、耦合最少" 这两者部分冲突，因为交易会话是**登录时协商出来的连接密钥**，绑在客户端进程里。一个完全独立的脚本要自己查持仓，只有两条路：

- **A1：脚本自己登录**。需要账号、密码、以及 `TJyaid.dll::GetXUserHardInfo` 生成的那套**硬件指纹**（绑机用的，`jyverify.dat` 就是它）。脚本独立跑，零耦合，但要逆登录流 + 硬件指纹算法 + 报文加密。
- **A2：脚本借会话**。客户端登录后，脚本从客户端进程内存里**一次性**读出连接密钥/会话 token，之后脚本独立讲协议、独立查持仓。耦合仅限于"读一次内存"，脚本本身独立运行。

A1 是终极目标但工作量最大；A2 是务实折中，能很快跑起来，之后可逐步往 A1 演进。

## "脚本主动发请求"带来的真问题

脚本主动发请求有两条实现方式，安全性取决于协议本身（我们还没逆出来）：

1. **借用客户端的 TCP 连接发请求**：客户端和脚本共用一条 socket。如果 7703 协议的包**带请求 ID 能配对**（像 CTP 那样），脚本用自己的 ID 发、自己收，互不干扰——可行。如果协议是**严格一问一答、无 ID**，脚本插一脚会让客户端的状态机错乱（拿到脚本的响应当成自己的）——不可行。
2. **脚本自己开连接、自己登录**：复用从客户端抓到的凭据 + `TJyaid.dll::GetXUserHardInfo` 的硬件指纹。但指纹绑机，脚本得在**同一台 Wine 环境**里调用 `TJyaid.dll` 生成指纹（这算轻度耦合），且服务器可能限制单账号并发会话。

到底走哪条，得等抓到真实包、看到协议结构（有没有请求 ID、登录握手形态）才能定。**所以第一步必然是：抓 7703 流量。**

## 唯一可行的架构：进程内薄垫片（shim）

既然密钥只能在客户端进程里活着，那就**借进程**，但只借一点点：

```
TdxW.exe (Wine)
  └─ 注入 shim.dll（极小）
       ├─ hook GM-SSL 记录层的 encrypt/decrypt
       └─ 暴露一个本地 Unix socket：明文收发接口
            ▲
            │ 明文
            │
独立 Linux 脚本 ── 构造「持仓查询」明文请求、解析明文响应
```

- **shim 极小、只做一件事**：把客户端解密后的明文转给脚本，把脚本的明文塞回客户端加密层。它复用客户端已协商的 SM4 会话，自己不做任何密码学。
- **脚本独立**：所有业务逻辑（持仓查询的明文结构、字段解析、主动发请求的时机）都在 Linux 侧脚本里，跟客户端 UI 完全无关。
- **耦合点只有一个**：注入一个 shim DLL。这是 GM-SSL 强制的最小耦合，没法更少了。

这条路要继续，下一步是定位 `TdxW.exe` 里的 GM-SSL 记录层函数（encrypt/decrypt 入口），靠字符串 `tdxgw_gmssl` 交叉引用能很快找到——这个字符串在握手明文里出现，说明客户端代码里一定有它做协议标识。

## 说明

- **数据流**：读客户端内存持仓 → 一次性批量取所有持仓的 5档 → 算每只的 micro price → 算 6 档名义金额滑点 → 打表。无 argparse。
- **micro price** = `(bid1×ask量 + ask1×bid量)/(bid量+ask量)`，标准加权微观价。
- **滑点** = `(成交均价 − micro)/micro`，单位 bp（基点）。买为正（吃卖盘越买越贵），卖为负（砸买盘越卖越贱）。
- **`>` 标记**：5档吃不满该名义金额，按已成交部分算的滑点是下界，实际更差，故加 `>`。例如 `605567 10W买 >+38.1bp`。
- **N/A**：无行情（如 `787825 长鑫申购` 是申购代码，没盘口）。

现在凌晨，是收盘快照——所以你会看到很多票 1W/5W/10W 买滑点相同（10万也只够吃到 ask1，因为收盘 ask1 挂单量大）。交易时段盘口薄、逐档吃深，不同名义金额的滑点会拉开差距，`>` 也会更多。开盘后再跑一次就能看到真实分层。
